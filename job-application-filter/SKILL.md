# 求职岗位筛选与投递优先级排序

## Skill 目标

将用户的简历与多个JD进行系统性匹配分析，输出：
1. 硬性过滤结果（直接排除不能投的）
2. 每个岗位的匹配度评估
3. 最终投递优先级排序
4. 每个岗位的风险点与策略建议

---

## 第一步：提取用户画像

从简历中提取以下信息（如简历未提供则询问用户）：

```
【基本信息】
- 毕业时间（精确到月份，判断届次）
- 学历层次（本科/硕士/博士）
- 所在城市 / 可接受城市

【专业背景】
- 本科专业
- 研究生专业（如有）
- 相关课程

【实习经历】
- 公司类型（互联网/创业/传统）
- 岗位性质（产品/运营/研发/数据等）
- 核心关键词（PRD/A/B Test/用户研究/数据分析等）

【项目经历】
- 项目方向（AI/电商/内容/工具等）
- 核心产出（报告/原型/模型/系统等）

【技能】
- 编程语言
- 产品工具（Figma/PRD/SQL等）
- AI工具使用深度

【语言能力】
- 英语水平（流利/良好/基础）
- 其他语言

【特殊加分项】
- 海外留学/生活经历
- 有无内推码（影响机筛策略）
- 可实习时长/每周天数
```

---

## 第二步：逐岗硬性过滤

对每个JD检查以下硬性条件，**任一不满足则标记为❌直接排除**：

| 过滤项 | 检查逻辑 |
|---|---|
| 毕业届次 | ByteIntern/秋储通常要求特定届次，确认用户毕业时间是否在范围内 |
| 实习时长 | JD要求的最短实习期 vs 用户实际可用时间 |
| 每周天数 | JD要求天数 vs 用户课程安排 |
| 专业硬排除 | 少数岗位会明确排除某专业（如"新闻/心理/电影专业优先"的AI游戏岗对CS背景是减分） |
| 特殊硬要求 | 如"需驻场XX城市"、"需要XX语言证书"等 |

---

## 第三步：岗位类型价值评级

```
S级：ByteIntern / 秋储转正 / 留用实习（有明确转正通道）
A级：日常实习（知名大厂，背书价值高）
B级：日常实习（中型公司或细分领域）
C级：普通实习（无明显品牌价值）
```

**注意：** 有内推码时，S级岗位优先级进一步提升，因为机筛不再是瓶颈。

---

## 第四步：匹配度评分（满分10分）

对每个通过硬性过滤的岗位，从以下5个维度打分：

### 4.1 关键词命中率（机筛逻辑）——权重30%

检查简历中能直接命中JD关键词的比例：

**高价值命中词（每个+1分）：**
- 专业背景直接匹配（如JD写"计算机优先"，用户是CS专业）
- 技能栏直接列出的工具（SQL/PRD/Figma/A/B Test等）
- 实习经历中出现的核心场景词（用户增长/数据分析/产品迭代等）
- 项目经历中的技术词（AI/机器学习/大模型等）

**降权场景：**
- JD明确写"XX专业优先"但用户专业不符
- 有明确经验要求但用户完全没有相关背景

> **有内推码时：** 此维度权重降低至10%，重点转向4.2和4.3

### 4.2 面试竞争力（人工筛逻辑）——权重35%

评估用户在该岗位的差异化竞争力：

**加分项：**
- 海外背景（对国际化/海外产品岗是稀缺竞争力）
- 英语流利（对海外岗/跨境电商岗是核心要求）
- 同类场景实习经验（如Yahaha游戏经历对游戏AI岗）
- AI工具深度使用（对AI产品岗加分）
- 完整产品链路经验（从用户调研到PRD到上线）

**减分项：**
- 完全没有对口行业经验（如零金融背景申请金融产品岗）
- 岗位强调ToB但用户只有ToC经验
- 岗位有特殊专业偏好但用户专业不符

### 4.3 故事线完整度——权重20%

用户的经历能否形成一条清晰的叙事：

- **强故事线**：经历→项目→技能形成完整闭环，面试时有具体案例支撑
- **弱故事线**：经历和岗位方向跨度大，需要大量包装

### 4.4 特殊申请要求——权重10%

- 是否有额外材料要求（cover letter / 三条bullet / portfolio）
- 有额外要求的岗位需要额外准备时间，影响投递效率

### 4.5 岗位稀缺性——权重5%

- JD是否明确写"简历稀缺"（如快手策略产品北京场次）
- 是否对特定背景有明显偏好（如英语+海外背景）
- 岗位是否刚发布（越新越容易被看到）

---

## 第五步：输出格式

### 5.1 总览表格

```
| 优先级 | 岗位名称 | 公司 | 岗位类型 | 匹配分 | 核心理由 | 主要风险 |
|--------|---------|------|---------|--------|---------|---------|
| 🥇 | ... | ... | S级 | 9/10 | ... | ... |
```

### 5.2 逐岗详细分析

对每个岗位输出：
- ✅ 命中的JD要求（列出对应简历内容）
- ⚠️ 弱项/风险点
- 💡 投递策略建议（如需要包装哪个经历、是否需要额外材料）

### 5.3 最终投递建议

```
【冲】优先投递，匹配度最高或转正价值最大
【稳】保底，机筛稳定或人工筛有竞争力
【补】日常实习积累经验，随时可开始

有投递数量限制时：给出具体志愿顺序
有内推码时：重新排序，冲击匹配度最高的岗位
```

---

## 特殊场景处理

### 场景A：有内推码
- 机筛关键词权重大幅降低
- 优先考虑"故事线最完整"的岗位，而非"最安全"的岗位
- 可以冲击专业不完全匹配但经历高度对口的岗位

### 场景B：有投递数量/顺序限制（如美团"最多3个，第一志愿优先"）
- 第一志愿：匹配度最高的岗位（即使竞争激烈）
- 第二志愿：匹配度次高但更稳健的岗位
- 第三志愿：保底或转正价值高的备选

### 场景C：同公司多岗位
- 优先考虑同一公司内匹配度最高的一个深度投递
- 避免简历在同公司内因为多岗位投递被HR标记

### 场景D：跨城市投递
- 询问用户是否有城市偏好
- 如用户说"地点不重要"则忽略此维度
- 如涉及香港学生去深圳通勤，默认认为可行

---

## 注意事项

1. **届次判断要精确**：ByteIntern通常指"2026年9月至2027年8月毕业"，需核对用户的精确毕业时间
2. **包装 ≠ 造假**：建议用户包装是指用合适角度描述真实经历，不要建议虚构经历
3. **日常实习无届次限制**：不要把届次要求套用到日常实习岗位
4. **告知额外申请要求**：如KAWO需要三条bullet，需提醒用户这比简历本身更重要
5. **机筛 vs 人工筛的不同逻辑**：有内推码时两套逻辑完全不同，要明确告知用户
